# Plan: Fix Mobile Chess Piece Display Bug

## Problem
On mobile browsers, black pawn pieces (and possibly other black pieces) display incorrectly - showing "giant looking and mismatching style" instead of the correct Unicode chess glyphs. This issue doesn't reproduce on desktop even with Chrome dev tools mobile emulation.

## Root Cause Analysis

### Key Finding
The chess pieces are rendered as Unicode characters (e.g., `♟` for black pawn) in `BoardView.js`:

```javascript
const PIECE_TO_GLYPH = {
  wP: "♙", wN: "♘", wB: "♗", wR: "♖", wQ: "♕", wK: "♔",
  bP: "♟", bN: "♞", bB: "♝", bR: "♜", bQ: "♛", bK: "♚",
};
```

### Likely Causes (in order of probability)

1. **Font Fallback Issue on Mobile**: 
   - The CSS in `layout.css:21` uses `clamp(0.875rem, 8cqmin, 3.25rem)` with a fallback for browsers not supporting `cqmin`
   - Mobile browsers may have different Unicode chess character support, falling back to system fonts that render inconsistently

2. **Missing Font-Family for Pieces**:
   - The `.chess-piece` class (`layout.css:91`) doesn't specify a font-family, relying on inheritance
   - On mobile, the inherited font may not include chess Unicode characters, causing fallback to a different (and larger) glyph

3. **Vanduo's text-size-adjust:none**:
   - `dist/vanduo.min.css` sets `text-size-adjust: none` which can cause font rendering issues on mobile

4. **Container Query (cqmin) Support**:
   - The `8cqmin` unit may behave differently on actual mobile browsers vs desktop emulation

## Solution

### Fix 1: Explicit Font-Family for Chess Pieces
Add explicit font-family to `.chess-piece` in `styles/layout.css`:
```css
.chess-piece {
  /* ... existing styles ... */
  font-family: "Segoe UI", "Apple Color Emoji", "Noto Color Emoji", "Symbola", "Arial Unicode MS", monospace;
}
```

### Fix 2: Mobile-specific font-size adjustment
Add mobile-specific rules with more conservative sizing and explicit font settings

### Fix 3: Add Font-Display Swap
Ensure fonts load properly on mobile
