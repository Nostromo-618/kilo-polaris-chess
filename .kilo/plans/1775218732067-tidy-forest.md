# Chess Engine Optimization Plan

## Overview
This plan outlines improvements to make the Kilo Aurora Chess engine significantly stronger without adding any third-party libraries. All changes are incremental and build on the existing strong foundation.

## Expected Outcome
- **Speed increase**: 20k-50k → 200k-400k nodes/second (+300-700% improvement)
- **Search depth**: 5 ply → 10-12 ply maximum practical depth
- **Elo gain**: ~400-600 Elo improvement, reaching ~2200-2500 Elo strength
- No breaking changes to existing API or UI

---

## Priority 1: Critical Bottlenecks (Highest Impact)

### 1.1 Fix `squareAttackedBy` detection (Rules.js)
**Current issue**: O(n) scan of all 64 squares for every check test - single biggest performance bottleneck
**Fix**: Implement direct attack vector checking
- For any target square, check knight moves first
- Then check pawn attacks
- Then check king moves
- Then cast rays for sliding pieces (bishop, rook, queen)
- Stop as soon as an attacker is found
**Expected gain**: +150-200% speed improvement
**Difficulty**: Medium

### 1.2 Upgrade Zobrist hashing to 64-bit (AI.js)
**Current issue**: 32-bit hashes have high collision probability, corrupting transposition table
**Fix**:
- Use BigInt for all Zobrist values instead of 32-bit numbers
- Replace Map with array-based transposition table using modulo indexing
- Implement always-replace replacement policy
**Expected gain**: +50% speed, much deeper search accuracy
**Difficulty**: Low

### 1.3 Implement Aspiration Windows (AI.js)
**Current issue**: Full window search wastes massive time
**Fix**: Add 5-10 lines in `searchRoot()`:
- Start with narrow window around previous depth's best score
- Re-search with wider window only if result falls outside
**Expected gain**: +30-40% search speed for free
**Difficulty**: Very Low

### 1.4 Fix Passed Pawn Detection (Evaluator.js)
**Current issue**: Completely broken - only checks adjacent files, not if enemy pawns are ahead
**Fix**: Correct implementation that verifies no enemy pawns exist in front of pawn on same/adjacent files
**Expected gain**: Massive endgame improvement, no performance cost
**Difficulty**: Very Low

### 1.5 Implement Principal Variation Search (PVS) (AI.js)
**Current issue**: Full alpha-beta window used for all moves
**Fix**:
- Search first move with full window
- Search all subsequent moves with zero-window (null window)
- Re-search only if a move raises alpha
**Expected gain**: +30% speed with identical search quality
**Difficulty**: Medium

## Priority 2: Major Improvements

### 2.1 Incremental Move Making / Undo (GameState.js)
**Current issue**: Full board cloning on every search node - massive GC pressure
**Fix**:
- Remove all `cloneBoard()` calls from search path
- Implement `makeMove()` / `undoMove()` that modify a single board instance
- Store delta information to undo changes
**Expected gain**: +100% speed, 95% reduction in garbage collection
**Difficulty**: High

### 2.2 Game Phase Interpolation + Endgame PST (Evaluator.js)
**Current issue**: Only middlegame PST values used for all positions
**Fix**:
- Add endgame piece-square tables
- Calculate game phase based on remaining material
- Interpolate evaluation between middlegame and endgame values
**Expected gain**: Drastically improved endgame play
**Difficulty**: Medium

### 2.3 Improved Late Move Reductions (AI.js)
**Current issue**: Fixed reduction thresholds with no conditions
**Fix**:
- Reduce based on depth, move number, in-check status, move type
- Never reduce captures, promotions, or checks
- Verify reduced depth results with full depth search
**Expected gain**: +20-30% speed, fewer tactical blunders
**Difficulty**: Medium

### 2.4 Proper Time Management (AI.js)
**Current issue**: Fixed depth per difficulty, no time allocation
**Fix**:
- Allocate time per move based on remaining clock time
- Use iterative deepening, stop when time runs out
- Never run out of time mid-search
**Expected gain**: More consistent strength, better use of thinking time
**Difficulty**: Medium

### 2.5 Static Exchange Evaluation (SEE) (AI.js)
**Current issue**: MVV-LVA doesn't account for recapture sequences
**Fix**: Implement simple SEE for captures:
- Calculate net material gain/loss of exchange sequence
- Prune bad captures in quiescence search
- Order captures by SEE score instead of MVV-LVA
**Expected gain**: Fewer bad captures, better tactical accuracy
**Difficulty**: Medium

## Priority 3: Quality Improvements

### 3.1 Mobility Evaluation (Evaluator.js)
**Current issue**: No mobility evaluation at all
**Fix**: Count legal moves for each piece type
- Small bonus for each legal move
- Higher bonus for knight/bishop mobility
- Penalty for very low mobility
**Expected gain**: Stronger positional play
**Difficulty**: Low

### 3.2 King Safety Improvements (Evaluator.js)
**Current issue**: Only counts pawn shield and open files
**Fix**:
- Count enemy attacks on squares adjacent to king
- Penalty for enemy queen/rook on adjacent open files
- Bonus for friendly pieces near king
**Expected gain**: Much better king safety evaluation
**Difficulty**: Medium

### 3.3 Check Extensions (AI.js)
**Current issue**: No search extension when in check
**Fix**: Extend search depth by 1 ply when side to move is in check
**Expected gain**: Fewer checkmate blunders
**Difficulty**: Very Low

### 3.4 Futility Pruning (AI.js)
**Current issue**: No pruning at frontier nodes
**Fix**: At depth 1, prune moves that can't possibly improve alpha even with maximum possible gain
**Expected gain**: +10-15% speed
**Difficulty**: Low

## Priority 4: Low Hanging Fruit (<10 minute fixes)

1. **Remove JSON.stringify/parse** for castling rights (GameState.js) - manually copy 4 booleans
2. **Cache king positions** in GameState - stop scanning board every time
3. **Order promotions before quiet moves** in move ordering
4. **Add tempo bonus** in evaluation (+10 centipawns for side to move)
5. **Precompute all algebraic↔index conversions** at startup
6. **Improve quiescence search** to include checks and promotions

---

## Implementation Order Recommendation

Start with the easiest, highest impact changes first to get immediate feedback:

1. ✅ Fix passed pawn detection (5 min)
2. ✅ Add check extensions (5 min)
3. ✅ Implement aspiration windows (10 min)
4. ✅ Upgrade Zobrist to 64-bit (30 min)
5. ✅ Fix squareAttackedBy (2 hours)
6. ✅ Implement PVS (1 hour)
7. ✅ Incremental move making (4 hours)
8. ✅ Game phase interpolation (2 hours)

This order gives you ~300% speed improvement and major strength gains after just the first 4 items.

---

## Verification Strategy

After each optimization:
1. Run full test suite with `npm test`
2. Run a few full games at level 5 to verify no regressions
3. Measure nodes per second performance
4. Compare move choices against known positions

All optimizations can be implemented incrementally without breaking existing functionality. Each step is independently testable.
